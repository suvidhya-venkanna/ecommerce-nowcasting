# Forecasting E-Commerce Sector Demand Using Public Search Trends and Macroeconomic Indicators

**Status:** Draft submission (capstone synopsis stage). Data pipeline and EDA are complete and use
real data. Model training (ARIMA, XGBoost, LSTM) is planned for the final submission and is **not**
included in this draft.

## Problem Statement

Official e-commerce sales statistics are published with a reporting lag of several weeks to months,
limiting their usefulness for timely decision-making by retailers, investors, and policymakers. This
project investigates whether publicly available, high-frequency data - internet search trends
(Google Trends) and macroeconomic indicators (inflation) - can be used to forecast ("nowcast")
Dutch e-commerce sector demand ahead of the official Eurostat release.

## Nowcasting Horizon

The official e-commerce turnover index for reference month T is published by Eurostat approximately
5-10 weeks after month T ends. This project aims to produce a usable estimate of that index at
**T+0** - the moment month T ends - using only information that would actually be available at that
moment:

| Source | Available at T+0? | What the model can actually use |
|---|---|---|
| Google Trends search index for month T | **Yes** (near real-time) | The current month's own value (used un-lagged) |
| HICP inflation for month T | Partially | The published monthly value, at its own natural lag |
| E-commerce turnover for month T (the target) | **No** (~5-10 week lag) | Lagged values of its own history only (t-1, t-3, t-12) |

Full lead-time detail for each source is documented in
[`data/data_dictionary.md`](data/data_dictionary.md).

## Objective

Predict the monthly Eurostat online/internet retail turnover index for the Netherlands (NACE Rev. 2
class G47.91) using search-trend and macroeconomic features, and evaluate:

1. Whether Google Trends search volume for "online shopping" is a statistically significant leading
   indicator for turnover (**RQ1**)
2. Whether adding HICP inflation improves forecast accuracy over a univariate time-series baseline
   (**RQ2**)
3. How non-linear ML models (Gradient Boosting / XGBoost) compare against a classical ARIMA baseline
   in handling non-linear shocks such as pandemic-era volatility (**RQ3**)
4. How much the nowcasting approach reduces the "information gap" between real-time market activity
   and the official statistical release (**RQ4**)

## How to Run

### 1. Install dependencies
```bash
pip install -r requirements.txt
```

### 2. Run the notebook
```bash
jupyter notebook notebooks/Ecommerce_Nowcasting_Capstone.ipynb
```
or open it in Google Colab (recommended - this is how it has actually been run and tested so far).
Running it end-to-end will:
- Pull real data from Eurostat (turnover, HICP inflation) and Google Trends (search interest)
- Save the raw API responses to `data/raw/`
- Clean, merge, and restrict the data to the window where all three sources have genuine (not
  back-filled) coverage
- Engineer lagged features for both the target and the search index
- Produce EDA visualizations, including a "ragged edge" chart and a cross-correlation plot for RQ1
- Save the model-ready dataset to `data/processed/full_dataset_clean.csv`
- Train and compare ARIMA, XGBoost (baseline, unregularized nowcasting, and regularized nowcasting
  variants), and an LSTM
- Run a dedicated structural-break check through the 2020 COVID-19 period (RQ3)
- Run a Diebold-Mariano significance test on the main model comparison
- Answer RQ4 by training separate direct multi-horizon models and converting the result into an
  explicit "weeks of information gap closed" figure

> **Note:** this notebook requires internet access (to call the Eurostat and Google Trends APIs,
> and to install `eurostat`/`pytrends` via the first cell). `pytrends` is an unofficial client and
> is subject to Google's rate limiting - if a request fails with a 429 error, wait a few minutes
> and retry.

## Status and a Real, Honest Finding

This notebook has been executed end-to-end in Colab with real data (through mid-2026). The
headline result from the most recent run is worth stating plainly rather than glossing over: the
full "nowcasting" XGBoost model (with search-trend features) performed **worse** than both a
simpler XGBoost baseline (no search features) and a plain ARIMA model, on an unregularized first
pass. This is most likely overfitting - the model is trained on roughly 72 monthly rows with 13
features, no regularization, and no cross-validation. The notebook now includes a regularized
XGBoost variant and a Diebold-Mariano significance test specifically to investigate this, rather
than reporting only the more favorable number. **This is treated as a genuine, reportable research
finding for the draft, not a bug to hide** - it directly motivates a discussion of sample size and
regularization in the final write-up, and is exactly the kind of result RQ1-RQ3's ablation and
significance testing are designed to surface.

## Planned Modeling Approach (Final Submission)

- **Baseline:** ARIMA - classical statistical time-series baseline, using only the target's own
  history
- **Main model:** XGBoost - using lagged target features, lagged and un-lagged search-index
  features, and HICP inflation; both an unregularized and a regularized variant are compared,
  given the overfitting risk noted above, to test RQ1-RQ3
- **Third model:** LSTM - to test whether a deep learning sequence model captures non-linear,
  shock-driven dynamics (e.g., pandemic-era volatility) differently than ARIMA or XGBoost
- **Structural break check:** ARIMA and XGBoost error explicitly compared during the 2020 COVID-19
  shock versus a typical, out-of-sample period, using models trained only on data that predates the
  shock - the main train/test split (train through 2024, test 2025+) never otherwise evaluates
  either model on the shock period at all, since it falls inside the training window
- **Evaluation:** RMSE, MAE, MAPE (all reported as percentages), and a Diebold-Mariano test for
  statistical significance of accuracy differences between models
- **RQ4 (information-gap quantification):** forecast accuracy at increasing horizons (trained as
  separate direct multi-horizon models, not a single model reused incorrectly across horizons) is
  converted into an explicit number of weeks of lead time gained, relative to Eurostat's documented
  publication lag (see the Nowcasting Horizon section above)

**Known limitations to address before final submission:**
- Only ~72 training months are available, constrained by Google Trends' historical coverage window
  (2018+) - this is the most likely driver of the overfitting result above
- ARIMA's order (1,1,1) is a fixed starting point, not selected via an AIC/BIC grid search
- The structural-break check (RQ3) trains on only ~12 months of pre-shock data, which is thin for a
  confident conclusion - a longer ARIMA-only history (built from the unrestricted raw Eurostat pull
  in `data/raw/`, since ARIMA does not need `search_index` at all) would strengthen this check
- The LSTM uses a fixed, untuned architecture (a single configuration, no hyperparameter search)

## Data Limitations

- **Google Trends measures intent, not realized purchases.** A search for "online shopping"
  reflects that someone was considering a purchase, not that a transaction occurred. This
  intent-vs-outcome gap is exactly what RQ1's leading-indicator test is designed to probe, not
  something assumed to hold uncritically.
- **Google Trends values are relative, not absolute**, and are normalized within the specific date
  range and region queried. This project fixes the query window (`START_DATE` to `TODAY`, `geo='NL'`)
  so values stay comparable across the study period, but they are still not a direct search-volume count.
- **The turnover series is seasonally and calendar adjusted (SCA) by Eurostat before this project
  receives it.** This means some of the seasonality a not-seasonally-adjusted series would show has
  already been smoothed out upstream - worth keeping in mind when interpreting the seasonal
  decomposition in the EDA notebook.
- **Structural breaks are a known risk.** The 2020 COVID-19 period produced a sharp, atypical shock
  to e-commerce demand that may not follow the relationships the model learns from calmer periods.
  The final submission's modeling notebook should evaluate model error separately during this shock
  window versus a typical period, rather than relying on one overall average that could hide
  shock-period failure - this is part of the motivation for RQ3 (ARIMA vs. ML performance on
  non-linear shocks).
- **This dataset covers the Netherlands only**, restricted to 2018 onward (see the data dictionary
  for why). Findings should not be assumed to generalize to other countries or to the pre-2018 period.
